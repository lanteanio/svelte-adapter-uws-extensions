## [Unreleased]

## [0.5.1] - 2026-05-17

### Fixed

- **`bus.wrap(platform)` now forwards `platform.redis` and `platform.presence` on the wrapped seam (live getters), matching the existing `replay` forwarding.** Pre-fix, only `replay` was forwarded. Framework code that stashes conventions on the source platform in init - e.g. `platform.redis = redis.redis` to enable `svelte-realtime`'s cluster-shared `live.room({ presence })` Redis HASH - found those conventions on the source platform but NOT on the wrapped platform that `createMessage({ platform: bus.wrap })` hands to per-message ctx. The framework's `_clusterPresenceAcquire` / `_clusterPresenceList` then saw `ctx.platform.redis === undefined` and fell through to the in-memory `_presenceRef` Map, silently negating the cluster-presence fix. The bug manifested in any multi-replica deploy that wired `platform.redis` correctly in init but still saw per-replica "Online" rosters in chat / room demos. The fix adds live getters for `redis` and `presence` to the wrapped object so post-wrap reassignment on the source platform also propagates - same pattern the existing `replay` getter uses.

## [0.5.0-next.16] - 2026-05-16

### Changed

- **`createPresence` storage layout rewritten for O(1) leave + per-field HEXPIRE TTL; requires Redis 7.4+.** Pre-fix, the per-topic hash stored one field per `(instanceId, userKey)` pair via compound field names (`{instanceId}|{userKey}`), and `LEAVE_SCRIPT` ran `HDEL my_field` followed by a Lua suffix-match scan over `HGETALL` of the topic hash to decide whether to broadcast a leave. At realistic auctions/chat scale (10k users x 10 instances = 100k presence fields per topic), a single leave was ~1-5 ms of Redis-event-loop-blocked Lua time. Mass disconnect of 1k users on instance restart amounted to ~1-5 seconds of presence-unavailable Redis. The new layout splits storage into two hashes per topic: `presence:topic:{topic}` keyed by `userKey` (backs `list()`/`count()`) and `presence:user:{topic}:{userKey}` keyed by `instanceId` (backs the leave HLEN check). `JOIN_SCRIPT` does HSET + HPEXPIRE on both atomically. `LEAVE_SCRIPT` does HDEL + HLEN; if HLEN drops to zero, HDEL the userKey from the per-topic hash and return 1 (broadcast leave). Mass-disconnect time is now O(N) total Redis-blocked Lua time regardless of M (the topic's total user count). Per-field TTL via `HPEXPIRE` replaces the heartbeat-driven CLEANUP_SCRIPT and the per-leave timestamp-filter scan: stale entries from crashed instances auto-expire field-by-field via Redis itself. The new layout also fixes `count()` correctness - the previous HLEN-based count returned stale-inclusive counts, while the new version is exact (per-field auto-expiry means HLEN reflects live users only).

  **Empirical results on real Redis 7.4 + ioredis** (head-to-head bench, M=10000 noise fields, N=1000 mass-disconnect users, both implementations on the same Redis instance with identical workload shape): pipelined `leaveAll` drops from **3654 ms (OLD) to 6 ms (NEW) - a 609x speedup**. Sequential variant drops from 4454 ms to 495 ms (9x; round-trip latency dominates the new case). Both correctly broadcast 1000/1000 leaves. The integration test `LEAVE_SCRIPT cross-instance behavior > the broadcast suppression scales to many cross-instance fields without scanning` pins the per-leave shape under noise at <100ms wall-clock with M=5000.

  **Migration shape.** Public API unchanged - `createPresence(redis, options)` takes the same `key` / `select` / `heartbeat` / `ttl` / `breaker` / `metrics` / `keyspaceNotifications` options and returns the same `RedisPresenceTracker` shape. Hash key names changed (old `presence:{topic}` -> new `presence:topic:{topic}` + `presence:user:{topic}:{userKey}`); old keys are ignored by the new code and expire naturally on Redis. During a rolling deploy with mixed old-and-new instances, list/count from new instances miss old-instance presence for one TTL window; subscribers see eventual convergence. The `metrics()` snapshot's `staleCleanedTotal` field is now always 0 (no application-side cleanup pass) but stays in the return shape for backward compatibility. The `presence_stale_cleaned_total` Prometheus counter is no longer registered. Redis minimum bumped from 7.0+ to **7.4+ for `createPresence` specifically** (other extensions unchanged); `createPresence` probes `INFO server` on first use and throws on older servers with a clear migration message. **9 new integration tests** in `test/integration/redis/presence.test.js` cover the new storage layout (per-topic + per-user hashes, per-field HPEXPIRE TTL, no whole-key TTL), the LEAVE_SCRIPT cross-instance HLEN check, mass-disconnect scaling under noise fields, per-field auto-expiry on real Redis timing, and heartbeat HPEXPIRE-only refresh (no HSET, which would clear TTL). The full mock suite (1433/1433) and integration suite (256/256) pass.

- **README: new `## Multi-tenant deployments` section in the Operations group documenting per-tenant Redis isolation for shared-Redis deployments.** Pre-fix, the README documented `keyPrefix` on the Redis client and per-extension key prefix sub-options but did not state in one place that ioredis's `keyPrefix` does NOT apply to pub/sub commands. A multi-tenant deployment that set `createRedisClient({ keyPrefix: 'tenant-A:' })` and assumed full isolation was still publishing on the literal channel `uws:pubsub` -- the same channel every other tenant on the same Redis was subscribed to. The new section names the two-namespace problem (keys vs. channels), enumerates the three extensions that default to a globally-named channel (`createPubSubBus`'s `channel: 'uws:pubsub'`, `createShardedBus`'s `channelPrefix: 'uws:sharded:'`, `createPublishRateAggregator`'s `channel: 'uws:pressure:rates'`) and shows the per-tenant override pattern for each, calls out the topic-name-collision shape on the sharded bus (one channel per topic, so topic naming travels through tenancy), covers the Postgres LISTEN/NOTIFY case (channel name is caller-supplied, must be tenant-segmented), and gives a full end-to-end multi-tenant example wiring (`tenantStack(tenantId)` factory + per-message `stackFor(ws.getUserData().tenantId)` routing). Closes with a costs note (per-tenant prefixes are operationally trivial; the bigger cost is operator discipline to apply them consistently). TOC entry added under Operations. No code change. documenting the `shared/sensitive.js` helpers' "safe to spread, safe to log" contract.** Pre-fix, `stripInternal`, `redactConnectionUrl`, and `createSensitiveWarner` were exported from `svelte-adapter-uws-extensions/sensitive` (per `package.json#exports`) but had no README presence. Callers either re-derived the spread/log safety guarantees from JSDoc or rebuilt their own redactor per call site. The new section enumerates each helper's contract explicitly: `stripInternal` is "safe to spread" (filters `__proto__` / `constructor` / `prototype` at every iteration depth) AND "safe to log" (drops keys matching `/token|secret|password|auth|session|cookie|jwt|credential/i`, replaces binary views with `'[bytes: <len>]'` so a `JSON.stringify(Buffer.from('jwt-token'))` cannot leak the credential bytes); `redactConnectionUrl` handles the three classes a naive `[^@]+@` regex misses (passwords containing `@`, IPv6 bracketed hosts, query-string passwords); `createSensitiveWarner` is the one-shot warner used by plugin `select` callbacks. Each helper section also names what the helper does NOT do (domain-specific PII, inbound payload validation, replacing not-logging-sensitive-data) so callers do not over-rely on the conservative substring pattern. TOC entry added under Observability. No code change. Pre-fix, the README documented each extension's mechanics (Lua scripts, sequence numbers, fence semantics) but did not name in one place that every extension is an authorization-free primitive whose only auth contract is "the caller has already decided". On a shared backend the failure modes are more pointed than on the in-memory variants: a wire-supplied lock key blocks the legitimate owner across the entire cluster, not just one process; an idempotency-key collision reads another tenant's cached commit; a NOTIFY trigger that does not encode tenancy fans out to every subscriber. The fix adds one canonical `## Authorization model` section between the client setup and the Redis-extensions block, naming the contract (handler is the gate, `upgrade()` provides identity, derive backend keys from trusted identity prefixes) with a concrete `idempotency.handle(...)` example, then adds a 1-3 line `> **Authorization:** ...` callout to each of the 14 named extension sections (Pub/sub bus, Sharded pub/sub bus, Replay buffer Redis, Presence, Rate limiting, Broadcast groups, Cursor, Replay buffer Postgres, LISTEN/NOTIFY bridge, Job queue, Idempotency store, Distributed lock, Distributed session, Task runner). Each callout explains the specific failure mode (tenant-key collision, subscribe gate, queue payload trust, task-runner arg trust, NOTIFY trigger as trust boundary) and links back to the canonical section. No code change; docs only.

### Added

- **Idempotency stores: `maxResultBytes` option + `IdempotencyResultTooLargeError`.** Pre-fix, both the Redis and Postgres idempotency stores accepted arbitrarily large `commit(result)` payloads. A handler that legitimately returned a 50 MB JSON blob would lock that blob into Redis / Postgres for the cache lifetime (default 48 hours), and the cache would reload it on every subsequent retry. Past a few hundred KB this becomes operationally meta-stable (Redis pubsub fragmentation, Postgres NOTIFY caps, bus envelope cap). The new `maxResultBytes` option (default 256 KB, matching the operational shape both backends handle cleanly) rejects oversized commits with a typed `IdempotencyResultTooLargeError` (`code: 'IDEMPOTENCY_RESULT_TOO_LARGE'`, `bytes`, `maxBytes`) BEFORE writing to storage, leaving the slot in its pending state. The caller can then `abort()` to release immediately, or let the `acquireTtl` (default 60s) expire it. The new error class is defined once in `shared/errors.js` and re-exported by both backend modules so consumer code is portable between them. Configurable per-store; pass `Infinity` to disable. 15 new regression tests in `test/redis/idempotency.test.js` and `test/postgres/idempotency.test.js`.

- **`monotonicNow()` exported from `shared/time.js` for lock / lease / timeout math.** Pre-fix, `redis/lock.js` measured elapsed time with `Date.now()` deltas. A backward NTP step or DST transition between successive reads made `elapsed` appear negative; the retry-budget check `if (elapsed >= callMaxWaitMs)` silently extends the timeout (sometimes indefinitely if the step is large) and the wait-time histogram observation can underflow its bucket boundaries. The new `monotonicNow()` is backed by `performance.now() + processStartEpoch` (snapshot taken at module load), so it returns a `Date.now()`-shaped number that advances strictly forward regardless of system-clock adjustments. `redis/lock.js` migrated to use `monotonicNow()` for the acquire-loop timing math (three call sites). The existing cached `now()` is unchanged - use it for wall-clock-shaped timestamps that go on the wire (~1s precision is fine). Rule of thumb: "how long did this take?" -> `monotonicNow()` deltas; "what time is it?" -> `now()`. JSDoc + module header documents both contracts. 9 new regression tests in `test/shared/time.test.js` cover the monotonic-advance contract, millisecond shape, and the cached-vs-uncached resolution contract.

- **`busValidator.acceptRaw(message)` computes byte length internally; `acceptSize(bytes)` deprecated.** Pre-fix, every bus subscriber pasted the same boilerplate before calling `acceptSize`: `const rawBytes = typeof message === 'string' ? Buffer.byteLength(message) : message.length;`. The caller-side computation was a foot-gun: passing `.length` (character count) on a UTF-8 string with multi-byte characters under-counted the byte length and the cap silently failed open. The new `acceptRaw(message)` takes the raw message (string / Buffer / Uint8Array / ArrayBufferView) and computes the byte length itself via `Buffer.byteLength` for strings and `.byteLength` for binary inputs. Non-string non-buffer inputs return `false` (fail-closed). All four in-tree bus subscribers (`redis/pubsub`, `redis/sharded-pubsub`, `redis/cursor`, `postgres/notify`) migrated to `acceptRaw`. The deprecated `acceptSize` API is retained for backward compatibility (third-party bus subscribers may rely on it) but marked `@deprecated` in JSDoc and TypeScript. 8 new regression tests in `test/shared/bus-validate.test.js` cover ASCII / Buffer / Uint8Array acceptance, multi-byte UTF-8 byte counting (the foot-gun the deprecated API enabled), cap boundary, and fail-closed handling of non-string non-buffer inputs.

- **`metrics.authedHandler(predicate)` opt-in auth wrapper for the `/metrics` endpoint.** Returns a uWS request handler that runs the caller-supplied predicate against `(res, req)` and serves metrics only when the predicate returns truthy. Async predicates are awaited; predicate exceptions are caught and treated as denial (no error info leaked back). Denials return `401 Unauthorized` with a minimal text body and no metrics payload. The default `metrics.handler` is unchanged - the recommended deployment shape remains "bind `/metrics` behind a network barrier" (private port, internal LB, sidecar scrape target); the helper exists for same-listener mounts where a network barrier is not available. README's "Mounting the endpoint" section updated to document both patterns. The wrapper uses `res.cork()` + `res.onAborted()` per uWS best practice so a client hang-up during an async predicate does not write to a dead socket. 9 new regression tests in `test/prometheus/metrics.test.js` covering allow / deny / async / predicate-throws / token-pattern / abort-during-predicate / arg passing / non-function-predicate rejection.

### Security

- **Prometheus histograms: `buckets` array length capped at 32 by default.** Pre-fix, an `m.histogram('foo', 'help', [], bucketArray)` registration accepted a bucket array of any length. An attacker-influenced bucket array (rare but reachable if a config value sourced from a request body / cookie / query string ever flows into a histogram registration) of N buckets multiplied each labelset's per-histogram state by O(N) and each scrape's wire output by O(N) per labelset. Beyond the DoS shape, the underlying design point matches mature client libraries: Go's `client_golang` warns past ~100 buckets, Java's Micrometer soft-caps at 64, and percentile-resolution gains past 32 well-chosen buckets are marginal while per-labelset state and prom-scrape volume grow linearly.

  Fix: histogram registration now rejects bucket arrays longer than 32 (the new default `maxBuckets`). Configurable both registry-wide (`createMetrics({ maxBuckets })`) and per-histogram (7th positional arg to `histogram()`); pass `Infinity` to disable. The error message names the actual length, the configured cap, and both override knobs. Existing histograms with the default 9-bucket `DEFAULT_BUCKETS` array (or the connection-metrics duration/message/byte buckets at 8/6/5 entries) are unaffected. Apps that legitimately use longer arrays opt up explicitly. Zero runtime cost - the check fires once at registration, not per observation.

- **Prometheus metrics: per-metric `maxSeries` cap with reject-on-overflow + warn-once.** Pre-fix, `createMetrics()` placed no cap on the number of (metric, labelset) combinations a single counter / gauge / histogram could accumulate. A caller labeling by an unbounded-cardinality dimension (raw client IP, message id, user-controlled topic without `mapTopic` containment) could grow a single metric's `samples` Map without bound, blowing process memory and producing a `/metrics` body large enough to time out scrapes. Even short of OOM, ~10k label combinations per metric makes Grafana queries time out and prom-scrape volume balloons; the metric becomes operationally useless before it actually breaks anything.

  Fix: every `Counter` / `Gauge` / `Histogram` instance now carries a `maxSeries` cap defaulting to 10000. Past the cap, new labelsets are silently dropped (the data-plane operation is a no-op rather than throwing - throwing on metric increment would crash apps), a one-shot `console.warn` fires per metric naming the cap and pointing at `mapTopic` as the typical fix, and a built-in `prometheus_series_dropped_total{metric=<name>}` counter (lazily registered on first drop so a healthy registry never adds the extra series to `/metrics`) tracks ongoing drops for alerting. Existing label combinations continue to increment normally past the cap. The cap is configurable both registry-wide (`createMetrics({ maxSeries })`) and per-metric (4th positional arg to `counter`/`gauge`, 6th to `histogram`); pass `Infinity` to disable. The dropped-series counter itself has `maxSeries: Infinity` so a high-cardinality drop pattern cannot make the drop counter start dropping. Performance: one `Map.size >= maxSeries` comparison per inc/set/dec/observe (one register read, one branch); unmeasurable on the data plane.

  Operator action: zero for apps under the 10k cap (effectively all production usage). Apps approaching the cap will see the one-shot warn directing them to bump `maxSeries` or apply `mapTopic` containment. Apps that intentionally use high-cardinality metrics can opt out per-metric via `m.counter('foo', 'help', ['label'], 50_000)` or registry-wide via `createMetrics({ maxSeries: 50_000 })`.

### Changed

- **`shared/assert.js` test-mode detection now per-call instead of module-load snapshot.** Pre-fix, `IS_TEST_MODE` and `IS_PROD` were `const` evaluated once at module import. A test that flipped `process.env.NODE_ENV` or `process.env.VITEST` partway through its run got the stale mode on every subsequent `assert()` / `devAssert()` call. Vitest setup files that re-enter mode mid-run (e.g. running a production-shape integration test inside an otherwise test-mode harness) couldn't drive the assert path through both modes. Fix: `isTestMode()` and `isProd()` are now functions that re-read `process.env` on each call. No API change. Cost is two property reads per assertion vs zero; the assertion path is not on the per-message hot path so this is unmeasurable. 2 new regression tests in `test/shared/assert.test.js` flip `NODE_ENV` between calls and assert that the second call observes the new mode.

- **`postgres/replay.js` `clear()` and `clearTopic()` now run their two DELETEs inside a BEGIN/COMMIT transaction.** Pre-fix, the two DELETEs (one against the data table, one against the seqTable) were sent as separate `client.query` calls. With a `pg.Pool` wrapper, each call may even check out a different connection - so a crash, network blip, or external interruption between them could leave the seqTable populated while the data rows were already gone, producing readers that see a non-zero `seq()` for a topic with no buffered messages. The new `shared/pg-tx.js` helper exports `withTransaction(client, fn)` which pins a single pooled connection, issues BEGIN, runs `fn(tx)`, then COMMITs (or ROLLBACKs + re-throws on error). Both replay-clear paths now use it. The testing mock (`testing/mock-pg.js`) gained a `pool.connect()` stub that returns the same query dispatch behind a release-noop wrapper; BEGIN/COMMIT/ROLLBACK are accepted as no-ops by the dispatch since the mock does not model Postgres-side atomicity (integration tier is the right place for that assertion). 7 new tests across `test/shared/pg-tx.test.js` and `test/postgres/replay.test.js` cover the happy path, ROLLBACK on inner throw, ROLLBACK on query error, best-effort release when ROLLBACK itself throws, and BEGIN/COMMIT presence in the replay clear paths.

- **`redis/registry.js` `setEntry` now wraps HSET + (optional HDEL) + EXPIRE in a single MULTI/EXEC.** Pre-fix, the three writes were three awaited round-trips. If the process died (or the Redis socket dropped) between the HSET and the EXPIRE, the key was left with the new fields but no TTL - a stale entry lingering past the presence-expiry window. The MULTI/EXEC wraps the writes in a Redis transaction so either all three land or none do, and collapses the round-trip count to 1. Win is mostly correctness: registry registration is not on the per-message hot path, so the round-trip saving is operational rather than throughput-critical. 1 new regression test in `test/redis/registry.test.js` spies on `redis.multi()` and asserts the HSET + EXPIRE both land in the transaction. The mock testing helper gained a `multi()` method (aliased to `pipeline()` since the mock does not model Redis-side atomicity; integration tests are the right tier for that assertion).

- **`redis/lock.js` heartbeat now chain-promises onto the previous in-flight refresh instead of `setInterval(async)`.** Pre-fix, the heartbeat used `setInterval(async () => { ... await redis.eval(...) })`. When a heartbeat eval took longer than `heartbeatMs` (slow Redis, network blip), each subsequent interval fire issued ANOTHER concurrent eval against the same fence key, racing the breaker accounting and breaking the "one in-flight refresh at a time" invariant the lease state machine relies on. The leader-election helper in `redis/leader.js` already used the chain-promises pattern; the lock plugin now mirrors it: `setInterval(() => { inFlight = inFlight.then(heartbeatTick) }, heartbeatMs)`. Concurrent in-flight heartbeats are now always 0 or 1, regardless of eval duration. Zero cost when heartbeats finish within the interval (the common case); when they don't, queued ticks execute serially via the promise chain. 1 new regression test in `test/redis/lock.test.js` drives a stubbed eval that takes 40ms with `heartbeatMs=5` and asserts max concurrent heartbeat evals stays at 1.

- **`redis/lock.js` retry sleep now uses +/-25% jitter to avoid thundering-herd handoffs.** Pre-fix, the acquire loop slept a fixed `retryDelayMs` between SET NX attempts. With N callers contending on the same key, every lock-handoff produced an N-way SET NX burst on the Redis socket - one winner, N-1 losers, all retrying again at the next `retryDelayMs` tick. The pattern wastes Redis CPU and produces a stair-step latency distribution. Fix: each retry sleep is now `retryDelayMs +/- 25%` randomized (`(Math.random() - 0.5) * 0.5 * retryDelayMs`), preserving the expected wait but smearing actual delays over `[0.75x, 1.25x]` of the configured base. Math.random is the right primitive (thundering-herd avoidance, not security-relevant; marker comment added inline). No public-API change. `+2 new test cases` in `test/redis/lock.test.js` covering (a) jittered sleeps are non-identical and bounded by the 1.25x ceiling, (b) the existing `maxWaitMs` budget clamp still applies under jitter.

### Fixed

- **`createNotifyBridge` validates the `channel` option against the same regex used for table names.** Pre-fix, the channel name was passed through a double-quote-escape (`"<name with "" doubled>"`) which is RFC-correct for Postgres delimited identifiers, but the regex constraint that table names get (`[a-zA-Z_][a-zA-Z0-9_]*` plus reserved-namespace rejection) was missing for channels. A misuse could still pass non-ASCII or reserved-prefix names through to LISTEN/NOTIFY queries. Fix: factory now rejects channel names that fail the same regex OR sit under `pg_*` / `information_schema*` namespaces, throwing at construction time before any query goes out. `+3 new test cases` covering shape rejection (spaces, dashes, dots, leading digits, embedded quotes), reserved-namespace rejection, and well-formed acceptance.

- **Connection registry `bootstrapIndex()` now pipelines HGETALLs and iterates Cluster masters.** Pre-fix, the bootstrap loop in `redis/registry.js` ran `await redis.hgetall(key)` once per scanned key - O(N) round-trips for N registered users (default SCAN COUNT is 100, so 100 keys = 100 round trips). On Cluster, the same SCAN-routes-to-one-node issue from the 6.1 fix applied: bootstrap silently saw only one master's worth of users. Fix: detect Cluster via `typeof redis.nodes === 'function'`, iterate every master independently, and pipeline HGETALLs per SCAN page (one `pipeline().exec()` per batch instead of N awaited `hgetall()` calls). The seen-key dedup set is preserved across master iterations. Standalone path unchanged in observable behavior; the speedup is ~100x for cold bootstrap (1 round-trip per batch instead of 100). Cluster path now actually loads all users. `+1 new test case` in `test/redis/registry.test.js` covering the pipeline-construction shape (5 planted entries land in one pipeline.exec() with 5 hgetalls queued, not 5 separate calls).

- **`createWorkerPool` now rejects non-`file:` URLs for `worker.path` at registration time.** Pre-fix, the option accepted any string Node's dynamic `import()` would accept - and on the current LTS line, `import()` still accepts `data:` URLs (which execute inline JavaScript). If a developer ever populated `worker.path` from attacker-influenceable input (a config value sourced from a cookie, query string, header, or message body - unusual but not impossible), an attacker could supply `data:text/javascript,...` and run arbitrary code inside the worker thread, with full access to the worker's `parentPort` / `workerData` / process privileges. The fix adds a startup-time guard at `createWorkerPool` that rejects any `path` not starting with `file://` (or a `URL` instance whose `.protocol !== 'file:'`). Throws a clear error pointing at the canonical `new URL('./handler.js', import.meta.url)` pattern. Native binary URLs (`node:` builtins, `https:` already blocked by Node) are not the realistic surface here - this defense is specifically against `data:` execution. `+6 new test cases` in `test/postgres/tasks-worker-pool-isolation.test.js` covering data: rejection, http: rejection, bare-relative-path rejection, plus file: URL acceptance for both string and URL-instance forms.

- **`scanAndUnlink` (the `clear()` helper used by cursor / presence / replay / replay-stream / idempotency / ratelimit) is now Redis-Cluster-correct.** Pre-fix, the helper called `redis.scan(...)` directly. On a `Redis.Cluster` client, ioredis routes a keyless command like SCAN to one randomly-sampled node (`cluster/index.js#sendCommand` fallback to `getSampleInstance`); subsequent calls within the same loop may target different nodes, the cursor is meaningful only for the node that produced it, and keys owned by other nodes are silently skipped. Worse, the batched `redis.unlink(...keys)` is rejected with `CROSSSLOT Keys in request don't hash to the same slot` whenever the batched keys span hash slots - which they routinely do even within a single node's slot range. So any `clear()` call against a Cluster either dropped a random subset of keys silently OR threw a CROSSSLOT mid-loop. Fix: detect Cluster via the canonical `typeof redis.nodes === 'function'` API surface, enumerate `redis.nodes('master')`, and run an independent SCAN on each master. On Cluster, switch to per-key UNLINK (slower but correct); on standalone, the original batched UNLINK is preserved for throughput. `clear()` is an admin / test-reset operation so the per-key overhead is irrelevant. `+5 new test cases` in `test/shared/redis-scan.test.js` covering the standalone path (existing behavior preserved) and a mocked Cluster with three masters that explicitly throws CROSSSLOT on multi-key UNLINK (regression-proof against the audit's bug shape).

### Added

- **`ReplaySerializationError` exported from `postgres/replay`, `redis/replay`, and (via re-export through `redis/replay`) the stream backend.** Thrown when caller-supplied `data` cannot be serialized to JSON - typically a `BigInt`, a circular reference, or another value `JSON.stringify` refuses. Distinct from `ReplayStorageError` so callers can pattern-match: serialization failures are caller-input bugs (fix the payload), storage failures are operational (retry, fall back, alert). The original `TypeError` is preserved on `.cause`. Type declarations added to both `redis/replay.d.ts` and `postgres/replay.d.ts`.

### Fixed

- **Redis cursor and presence trackers now deep-strip the `select()` output before broadcasting, not just the default-select case.** Pre-fix, `safeUserData(ws)` did a shallow rest-spread `{ __subscriptions, remoteAddress, ...safeData } = raw` and passed the still-shallow result to `select`. The default `select` is `stripInternal` (which IS recursive), so the default-select path was always safe. But a user-supplied identity or partial-passthrough select - the documented pattern for surfacing additional userData fields, e.g. `select: (ud) => ({ id: ud.id, profile: ud.profile })` - would let nested sensitive keys (`profile.token`, `profile.authToken`, etc.) flow through to the wire unredacted. Fix: deep-strip the `select` output with `stripInternal` before storing on the per-ws state in cursor and before serializing for the presence diff. The `warnSensitive` call moved to run on the raw select output (so the developer-facing warning still fires on what their select returned, rather than silently swallowed by the backstop). resolveKey in presence continues to run on the pre-select shallow safeData so key resolution is unchanged. `+5 new test cases` across `test/redis/cursor.test.js` and `test/redis/presence.test.js` covering identity select with nested `token`/`authToken`, partial-passthrough select with nested `__internal`, and select that re-injects `constructor`/`prototype` (defense against shadow-via-Object.assign).

- **`publish()` and `publishIdempotent()` in all three replay backends now distinguish payload-serialization failures from storage failures.** Pre-fix, each backend called `JSON.stringify(data ?? null)` INSIDE the storage `try` block. A caller passing a `BigInt` (or any value `JSON.stringify` refuses to encode) triggered a synchronous `TypeError`, which the surrounding `catch` either (a) silently routed to `platform.publish(topic, event, data)` when `localFanoutOnStorageFailure: true` was set (silently degrading the durability contract on payloads the user thought were being persisted - the local fanout would either re-throw on its own serializer or land the message in subscribers without ever writing to durable storage), or (b) wrapped as a misleading `ReplayStorageError('publish', err)` suggesting Redis/Postgres was down when in fact the payload was malformed. Fix: hoist `JSON.stringify(data ?? null)` BEFORE the storage try-block in all four call sites (`postgres/replay.js#publish`, `redis/replay.js#publish`, `redis/replay-stream.js#publish`, `redis/replay-stream.js#publishIdempotent`); serialization failure throws the new `ReplaySerializationError` regardless of the `localFanoutOnStorageFailure` setting. No storage call is issued when serialization fails (so the idempotency slot is not consumed by a malformed retry). `+13 new test cases` across the three backend test files covering BigInt rejection, circular-reference rejection, the no-fallback-on-localFanout invariant, and verifying that no storage call (no `redis.eval`, no `client.query`) is issued when serialization fails.

- **`safeCreate(client, ddl, expectedColumns?)` now optionally verifies that an existing same-named table has the columns the swallowed CREATE TABLE was supposed to create.** Pre-fix, `safeCreate` silently swallowed `42P07` (relation already exists), so a pre-existing table with the same name but different schema would surface much later as a confusing `42703` (undefined_column) error from a downstream query. Fix: when the caller supplies `{ table, columns }`, the helper queries `information_schema.columns` after the swallow and throws with a clear error message listing the missing columns. Existing callers that omit the third argument retain the original silent-swallow behavior (backward compatible). The four `ensureTable()` sites in `postgres/_tasks-sql.js`, `postgres/idempotency.js`, `postgres/jobs.js`, and `postgres/replay.js` now pass `expectedColumns` for their primary CREATE TABLE statements. `request_id` is intentionally omitted from the jobs / tasks expected sets because it's added by the immediate follow-up `ALTER TABLE ADD COLUMN IF NOT EXISTS` for forward-migrated next.1 deployments; including it would false-positive on those tables. `+14 new test cases` in a new `test/shared/pg-migrate.test.js` covering assertSafeTableName + safeCreate + the new drift detection (verification runs on 42P07 with expectedColumns, throws on missing columns, accepts extra columns from forward-migration, no-ops on success / non-42P07 errors / omitted expectedColumns).
- **Prometheus `Histogram.observe()` now rejects negative values by default (matches `Counter.inc()` behavior).** Pre-fix, `assertFinite(value, 'histogram.observe()')` blocked NaN and Infinity but allowed negatives. Negative observations corrupt `_sum` (so `rate(metric_sum[5m])` returns nonsense) and break histogram bucket monotonicity (the bucket counts no longer satisfy `bucket[i] <= bucket[i+1]`, which downstream tooling assumes). Standard Prometheus convention is histograms-are-non-negative; the Counter class in the same module already enforces this with `throw new Error('counter value must not be negative')`, so Histogram's exemption was an inconsistency. Fix: throw `Error('histogram value must not be negative (pass allowNegative: true to opt in)')` when a negative observation is made without explicit opt-in. The factory function `metrics.histogram(name, help, labelNames?, buckets?, allowNegative?)` gains a 5th positional arg; the `.d.ts` describes the opt-in for signed-observation use cases (latency skew, P&L, temperature deltas). `+6 new test cases` covering negative-with-labels, negative-bare-overload, zero/positive still accepted, allowNegative opt-in works, NaN rejected regardless.

- **Redis replay backends (sorted-set + stream) now reject malformed `sinceSeq` values on `since()` and `replay()`.** Pre-fix, both backends had a ternary that fell through to "read everything" on negative input: sorted-set computed `sinceSeq + 1` as the ZRANGEBYSCORE lower bound (so `-1 + 1 = 0` returned every entry); stream computed `since >= 0 ? \`(${since}-0\` : '-'` (so `-1` became `-` which is XRANGE-from-start). NaN, Infinity, and fractional inputs slid through similarly. Authorization (the `H-RESUME-AUTH` checkSubscribe gate from next.5) blocks unauthorized callers entirely, so the leak only applies when an authorized caller is buggy or forwards client input unchecked - defense-in-depth, not a live exploit. Fix: both `since()` and `replay()` entry points in `redis/replay.js` (sorted-set) and `redis/replay-stream.js` (stream) now require `Number.isInteger(value) && value >= 0`. Invalid values cause `since()` to return `[]` and `replay()` to emit only an `end` marker. The `resumeHook` wire-boundary normalizer in both backends also tightened from `typeof === 'number' && >= 0` to `Number.isInteger() && >= 0` (fractional / NaN / Infinity now fall through to `0` (resume from start) instead of being passed verbatim to the backend). `+12 new test cases` in `test/redis/replay.test.js` and `test/redis/replay-stream.test.js` covering negative / NaN / Infinity / fractional / non-number sinceSeq for both backends.
- **Postgres worker pool now rejects IPC messages from a worker referencing an in-flight id that was dispatched to a different worker.** Pre-fix, the `w.on('message', ...)` handler in `postgres/_tasks-worker-pool.js` looked up `inFlight.get(msg.id)` and resolved/rejected the slot without verifying that the slot's `worker` field matched the worker emitting the message. The existing `w.on('error', ...)` handler in the same file ALREADY did this check (`if (slot.worker === w)` at line ~107); the message handler was the inconsistent path. A misbehaving or compromised handler file (the worker IS user code; the path is user-supplied via `runner.register('name', null, { worker: ... })`) could post `{type:'result', id:'<victim-id>', result:'hijacked'}` and the parent would resolve the victim's task with the poison value, AND the victim worker's eventual legitimate result would be silently dropped at the `if (!slot) return;` early-exit. Fix: one-line `if (slot.worker !== w) return;` after the slot lookup, mirroring the existing error-handler pattern. `+3 new test cases` in a new `test/postgres/tasks-worker-pool-isolation.test.js` that mocks `node:worker_threads` to exercise the (a) forged-result-from-wrong-worker, (b) forged-error-from-wrong-worker, and (c) unknown-id (pre-existing guard) paths.
- **`redactConnectionUrl` now correctly redacts passwords containing `@`, IPv6 hosts, and query-string `password` / `pass` / `pwd` parameters.** Pre-fix, the regex stopped at the first `@` after userinfo and leaked the tail of passwords containing unescaped `@` (libpq accepts `postgres://user:p@ssword@host` by greedy-matching to the last `@`); query-string `?password=...` connection params (libpq's alternate auth form) were not handled at all. The redactor is now a byte-level scan that (a) walks to the last `@` in the authority region, (b) suspends authority-terminator detection inside IPv6 `[...]` brackets, and (c) makes a second pass over the query region to replace `password=` / `pass=` / `pwd=` values (case-insensitive) up to the next `&` or fragment marker. `+11 new test cases` covering password-with-`@`, IPv6 hosts (`[::1]`, `[2001:db8::1]`), combined password-with-`@`+IPv6, query-string `password`/`pass`/`pwd` redaction, case-insensitivity, sibling-param preservation, simultaneous userinfo+query redaction, non-matching substring keys (`application_name`, `passthrough`), empty password (`user:@host`), fragment preservation, and `?` inside a fragment (not treated as query). The Postgres pool error handler (shipped above) benefits transparently. Public API contract unchanged; all 10 previously-pinned shapes still produce identical output.
- **Postgres idle-client pool error handler now redacts the connection password before logging.** Pre-fix, `pool.on('error', err => console.error('postgres: idle client error', err.message))` shipped the raw `err.message` to stderr; pg embeds the connection DSN in some failure-mode messages (auth failed, host unreachable, SSL mismatch), so a transient idle-client failure could log `connect ECONNREFUSED postgres://app:hunter2@db.internal:5432/main` to log aggregators verbatim. Fix: pipe `err.message` through the existing `redactConnectionUrl` helper (already used by the Redis factory for the same class of leak per next.5 M-CRED-LEAK), with a defensive `String(err)` fallback when `err.message` is not a string. No app-side change. This closes the follow-up the next.5 entry explicitly anticipated ("The pg-side `err.message` may still surface the URL on some failure modes; that is a follow-up if it surfaces in a real log.").
- **`stripInternal` no longer leaks raw bytes from `Buffer` / `TypedArray` / `DataView` / `ArrayBuffer` values into log lines.** Pre-fix, the helper treated binary views as plain objects and iterated their stringified numeric indices via `Object.keys`, so `JSON.stringify` of the result emitted the raw byte sequence (e.g. `{"payload":{"0":106,"1":119,"2":116,...}}` for a JWT-bearing Buffer). Any caller piping auth tokens, session bytes, or arbitrary binary user content through `stripInternal` for "safe logging" was producing the exact opposite of safe logging. The fix detects binary views via `ArrayBuffer.isView(obj) || obj instanceof ArrayBuffer` and substitutes a `"[bytes: <byteLength>]"` placeholder string. Plain-object recursion is unchanged.
- **`stripInternal` now also filters the `constructor` and `prototype` keys as defense-in-depth.** Pre-fix only the `__`-prefix filter ran, which DID catch `__proto__` (the classic prototype-pollution vector for JSON.parsed payloads). Adding `constructor` and `prototype` to the filter blocks shadow-via-`Object.assign` shenanigans where a caller does `Object.assign(target, stripInternal(userData))` and the result shadows `target.constructor` (which downstream code may consult for `new target.constructor()` patterns). Low severity, zero cost. Sensitive-key tests in `test/shared/sensitive.test.js` extended to cover Buffer/Uint8Array/Int32Array/Float64Array/DataView/ArrayBuffer substitution, `__proto__` (own enumerable from `JSON.parse`) rejection, and `constructor` / `prototype` rejection. JSDoc on `stripInternal` (both `.js` and `.d.ts`) updated to document the new binary-view substitution and the spread-safety contract. `+7 tests` (was 11, now 18).

## [0.5.0-next.11] - 2026-05-10

### Security

- **`allowSystemTopics` default flipped from `true` to `false` on every bus subscriber (`createPubSubBus`, `createShardedBus`, `createNotifyBridge`).** The wire-level `__`-subscribe gate (adapter side) blocks clients from receiving forged system-topic frames, but the bus republish path bypasses it: a foreign Redis / Postgres publisher can `PUBLISH` an envelope with `topic: "__signal:victim"` and have it forwarded to the local platform via `platform.publish`, which IS reachable by server-side subscribers (`live.signal()` is one). The comment claiming "tested for parity" is now backed by 12 new cases in `test/shared/bus-validate.test.js` covering the unconditional-rejection corpus plus the new `allowSystemTopics: false` default behavior. ## [0.5.0-next.10] - 2026-05-10

### Changed

- **Bumped `engines.node` to `>=22.0.0` (was `>=20.0.0`).** Tracks `svelte-adapter-uws` 0.5, which pins `uWebSockets.js` v20.67.0 (Node 20 dropped upstream). Node 22 LTS, Node 24 current, and Node 26 supported. See `MIGRATION.md` for the runtime-bump checklist.
- **Refreshed `node_modules`.** No source change; the remaining low-severity advisories all trace to `cookie<0.7.0` via the transitive `@sveltejs/kit` peer chain (unfixable until kit publishes a release that bumps cookie).

### Security

- **Postgres factories reject `pg_*` and `information_schema*` table names (LOW).** Pre-fix, the `idempotency`, `jobs`, `replay`, and `tasks` factories validated table names with `/^[a-zA-Z_][a-zA-Z0-9_]*$/` - which legitimately matches valid identifier shapes but also matches reserved Postgres internals (`pg_class`, `pg_authid`, `information_schema_tables`). A misconfigured `table` option here would either permission-error or, worse, succeed against a catalog the caller never intended to touch. Fix: each factory now rejects table names that lowercase to a `pg_*` or `information_schema*` prefix with a clear "reserved Postgres schema" error.

### Security

- **Redis `ConnectionError` no longer leaks the URL password (MED).** Pre-fix, when `new Redis(url, ...)` threw (DNS failure, malformed URL, bad TLS config), the `ConnectionError` message included the raw connection string - which for any non-trivial deployment carries a password (`redis://:s3cret@redis.internal:6379`). The error message landed in stderr, in any error-tracker hook, in the SaaS log pipeline, and in 5 years of search-indexed log retention. Fix: a new shared helper `shared/sensitive.js#redactConnectionUrl(url)` rewrites the password segment to `***` before interpolation; the redis client factory now interpolates the redacted form into the error message. Other URL forms (no userinfo, no password, query string with `@`, path-segment colons) pass through untouched.

### Added

- **`shared/sensitive.js#redactConnectionUrl(url)` exported.** New helper for redacting passwords in `scheme://user:password@host[:port][/path]` connection URLs. Used by the redis client factory; available for any extension or app-level code that interpolates connection URLs into log lines or error messages.

### Security

- **Inbound bus envelopes are validated before republish (HIGH).** Pre-fix, every bus subscriber (`redis/pubsub`, `redis/sharded-pubsub`, `redis/cursor`, `postgres/notify`) blindly republished whatever JSON envelope arrived: any actor with bus write access (compromised co-tenant in a shared Redis, leaked ACL, exposed port, hostile DBA) could fan a single PUBLISH out to every subscriber on every cluster node. Fix: a new shared validator (`shared/bus-validate.js`) gates every inbound envelope on (a) raw byte size before JSON.parse runs, (b) topic shape (string, 1-256 chars, no control chars - mirrors `isValidWireTopic` in the adapter), (c) event shape (string, 1-256 chars), (d) optional `__`-prefix denylist with an explicit allowlist for the bus's own internal channels. New per-bus options:
  - `createPubSubBus({ ..., maxEnvelopeBytes?, allowSystemTopics? })`
  - `createShardedBus({ ..., maxEnvelopeBytes?, allowSystemTopics? })`
  - `createNotifyBridge({ ..., maxEnvelopeBytes?, allowSystemTopics? })`
  - `createCursor({ ..., maxEnvelopeBytes? })` (the cursor module always validates the inner topic against the `__` denylist; the wrapper `__cursor:` prefix is constructed by the module).

  Default `maxEnvelopeBytes` is 1 MB; default `allowSystemTopics` is `true` for backward compatibility, but the pubsub bus's own configured `systemChannel` (default `__realtime`) is in the explicit allowlist either way. Apps that publish only over user-space topics can pass `allowSystemTopics: false` for defense in depth on top of the wire-level subscribe gate (adapter side).

- **Replay backends consult `platform.checkSubscribe` before reading a topic's history (HIGH).** Pre-fix, every `replay()` implementation (Redis sorted-set, Redis streams, Postgres) trusted the client-supplied `lastSeenSeqs` map verbatim; an attacker could send `{type:'resume', lastSeenSeqs:{"private:room": 0}}` and read the entire replay buffer for a topic they could not subscribe to live. Fix: every `replay(ws, topic, sinceSeq, platform, reqId)` calls `platform.checkSubscribe(ws, topic)` first; on denial, sends a single `{event:'denied', data:{code:<denial>, reqId}}` frame on `__replay:{topic}` and returns without reading the buffer. The client store treats this similarly to `truncated`. Older adapters that lack `checkSubscribe` degrade to current behavior so a peerDep mismatch does not hard-fail.

- **Postgres `tasks.run({idempotencyKey})` namespaced by task name (HIGH).** Pre-fix, two tasks with the same client-supplied `idempotencyKey` shared one cache slot; running `tasks.run('publicTask', {idempotencyKey:'abc'})` after a privileged `tasks.run('privateTask', {idempotencyKey:'abc'})` returned the private task's cached result without ever invoking the public handler. Fix: the cache key passed to `idempotency.acquire(...)` is now `'task:' + name + ':' + idempotencyKey`. Both `run()` and `enqueue()` also reject `idempotencyKey` longer than 256 characters (was unbounded; the framework boundary now enforces a key-length cap before a hostile caller can plant megabyte-sized keys in the store). The stores (`redis/idempotency`, `postgres/idempotency`) cap their `acquire(key)` argument at 1024 chars as defense-in-depth so a buggy framework caller still gets an explicit error rather than a B-tree depth amplifier.

### Added

- **`shared/bus-validate.js`: `createBusValidator(options)` and `isValidBusTopic(topic)`.** Shared module used by every bus subscriber; documents the bus-injection attack model in the file header. Exposes `acceptSize(bytes)` and `acceptEnvelope(topic, event)` - two-stage validation so a hostile size DoS is rejected before JSON.parse runs.

### Changed

- **Cache-key shape change for `live.idempotent` and `tasks.run` is NOT backward compatible.** In-flight cached entries from before this release become invisible after deploy (the namespaced key does not match the old un-namespaced key). For Redis stores the old keys eventually TTL out; for Postgres stores the cleanup interval handles them. No migration is required.

### Security

- **Lua scripts reject non-numeric ARGV with a clean `redis.error_reply` instead of crashing (HIGH).** Production paths validate ARGV in JS before EVAL, but the audit's fuzzer found that bypassing the JS layer (a future plugin author who passes user-supplied data without validation, a direct `redis.eval` against the script literal, or a poisoned hash entry with a non-numeric `ts` field) would crash the script with `attempt to compare nil with number` or `attempt to perform arithmetic on field 'ts' (a string value)`. Defensive `tonumber(x) ~= nil` checks added at the top of every script (`shared/scripts.js` CLEANUP/COUNT/COUNT_DEDUP/LIST, `redis/groups.js` JOIN, `redis/presence.js` JOIN/LEAVE, `redis/ratelimit.js` CONSUME/BAN, `redis/replay.js` PUBLISH, `redis/replay-stream.js` IDMP_PUBLISH/PUBLISH). The `pcall(cjson.decode, ...)` + `tonumber(parsed.ts)` pattern (already used by JOIN_SCRIPT in groups) is now applied uniformly so a poisoned hash entry is treated as stale rather than crashing the iteration.

The recovery sweep on any live instance then reclaims the row and re-drives the handler under the registered retry policy. Returns `true` if a row was running and got taken over, `false` if the row is no longer running (already terminal, never existed, or somebody else expired it first). Schema coupling collapses - no more direct table/column references in caller code.
- **`onStateChange` callback option on `createTaskRunner`.** Local-worker observer fired AFTER each state-machine transition commits: `null -> pending` (enqueue insert), `null -> running` (run insert), `pending -> running` (dispatch claim), `running -> running` (retry rearm OR recovery sweep, with `attempt` bumped), `running -> committed`, `running -> failed`. The callback receives `{ taskId, name, oldStatus, newStatus, attempt, requestId, result?, error? }` - the runner already owns every transition; this surfaces what it knows. Errors thrown from the callback (sync or via promise rejection) are caught and logged with `console.warn`; they do NOT roll back the state machine. Listeners run on the same tick as the SQL commit but are awaited fire-and-forget; the runner never blocks on the listener. Cluster-wide fan-out is a separate concern - still wants `postgres/notify`. Subsumes "publish to a realtime topic", "increment a metric", "log to stdout", "fire a webhook" without forcing every consumer to wire NOTIFY/LISTEN.
- **`createPgClient({ pool })` overload.** Wrap an existing `pg.Pool` instead of constructing a new one. Use when your app already maintains a pool (raw `pg` use elsewhere, another framework integration) and you want a single connection footprint against the database. When `pool` is provided: `connectionString` is no longer required, `autoShutdown` defaults to `false` (the caller owns the pool's lifecycle), `end()` becomes a no-op so graceful-shutdown paths can call `client.end()` unconditionally without worrying about ownership, and `pool.on('error')` is not installed (the caller's pool already has its handlers). Pass `connectionString` alongside `pool` if you also need `createClient()` (a dedicated `pg.Client` for LISTEN/NOTIFY) - the connection string is used only for that path. Existing callers that pass `connectionString` are unchanged.

### Breaking Changes

- **`redis/presence` wire shape on `__presence:{topic}` migrated to `presence_state` / `presence_diff` / `heartbeat`** (was `list` / `join` / `leave` / `updated` / `heartbeat`). Mirrors the adapter's bundled `createPresence` plugin so a single client decoder works for both single-instance and cluster deployments. New shape:

Same-tick joins+leaves on the same key collapse to the latest op; an "update" (re-join with new data) appears as a `joins` entry carrying the new data, since clients overwrite their `Map.set` on the same key.

  - **Public JS API on `createPresence` is unchanged.** `join()`, `leave()`, `sync()`, `list()`, `count()`, `metrics()`, `clear()`, `destroy()`, `hooks` keep their signatures and return shapes. The break is purely on the `__presence:{topic}` channel that clients decode.

Requires `svelte-adapter-uws >= 0.5.0-next.5`.
- **`wirePublishRateMetrics(platform, metrics, options?)` and `connectionMetricsHook(metrics, userClose?)` exported from `svelte-adapter-uws-extensions/prometheus`.** Two drop-in wirers for the adapter telemetry that lands on `platform.pressure.topPublishers` and the close-hook ctx (duration / messages / bytes). `wirePublishRateMetrics` registers `ws_topic_publish_rate` and `ws_topic_publish_bytes` gauges that pull from the pressure snapshot at scrape time via `gauge.collect()`, so there is zero accounting on the publish hot path. `connectionMetricsHook` returns a close-hook that emits per-connection histograms (`ws_connection_duration_seconds`, `ws_connection_messages_in/_out`, `ws_connection_bytes_in/_out`) plus a `ws_connection_close_total{code}` counter, and composes with a user-provided close hook by passing it as the second argument. Cardinality control: `topN` (default 10) on the publish-rate side, registry `mapTopic` honored on the gauges. Requires `svelte-adapter-uws >= 0.5.0-next.4`.

The captured id persists on the row through retry / recovery / dispatch sweeps and is exposed to handlers as `ctx.requestId` (task runner) or `job.requestId` (job claim), so handler-side logs from a different worker thread or instance can correlate back to the originating WS / HTTP request. Schema additions are `request_id TEXT` columns; existing 0.5.0-next.1 deployments forward-migrate transparently via idempotent `ALTER TABLE ADD COLUMN IF NOT EXISTS`. Empty-string `requestId` coerces to `null`. No metric labels carry the request id (UUID-shaped values are a cardinality bomb); persistence and ctx exposure only.

Callers who want custom truncation handling, batching, or other resume work alongside replay can compose by hand against `replay.replay(...)` directly. Requires `svelte-adapter-uws >= 0.5.0-next.4` for the `resume` hook.

## [0.5.0-next.1] - 2026-05-01

### Added

- **Idempotency store** (`svelte-adapter-uws-extensions/redis/idempotency` and `svelte-adapter-uws-extensions/postgres/idempotency`). Caches the result of an effectful operation under a stable key so retries within `ttl` return the original outcome rather than re-executing. Three-state `acquire(key)` returns `acquired` (caller runs the work, then `commit(result)` or `abort()`), `pending` (another caller is mid-flight), or `result` (cached). Two TTLs: `acquireTtl` (default 60s) bounds a pending slot to prevent crashed-owner deadlocks; `ttl` (default 48h) governs result cache lifetime. Redis backend uses a single-round-trip Lua script with `SET NX EX`. Postgres backend uses an atomic `INSERT ... ON CONFLICT DO UPDATE` with `expires_at` and a periodic cleanup sweep. Both backends share the same contract; the adapter's in-memory `createDedup` plugin is the zero-config single-instance fallback. Optional `breaker` and `metrics` options match the rest of the extensions.
- **Durable task runner** (`svelte-adapter-uws-extensions/postgres/tasks`). Wraps an effectful operation in a state machine that survives process crashes and naturally fans across cluster instances. `register(name, handler, { retry })` declares a task and its retry policy at registration time; `run(name, { input, idempotencyKey })` awaits the result. Three guarantees: caller-retry idempotency (pair with the idempotency store via the `idempotency` option to cache committed results), worker-crash recovery (every attempt holds a fence UUID; the conditional commit `UPDATE ... WHERE fence = $current` is atomic, and a periodic recovery sweep reclaims rows whose fence has expired), and external-service idempotency (the `idempotencyKey` is forwarded to the handler so it can be passed to Stripe / SendGrid / S3). Per-attempt heartbeat extends the fence while the handler is running; if the heartbeat fails, the handler's `AbortSignal` fires so it can bail gracefully. Per-handler retry config (`maxAttempts`, `backoff`, `on` predicate) ships in this entry; default is no retry on handler-thrown errors so non-idempotent tasks stay safe. Auto-creates a `svti_tasks` table; periodic cleanup deletes terminal rows past `rowTtl` (default 7 days). Optional `breaker` and `metrics` options.
- **Task runner worker-thread execution** (`worker` option on `register`). For CPU-bound handlers that would otherwise block the event loop, opt in by passing a URL/path to a separate file whose default export is the handler. The runner spawns a per-task thread pool (default size 1, idle timeout 30s, both configurable via `worker.pool`); each thread imports the handler file once and reuses it across runs. The `signal` in the handler context still fires on fence loss and is forwarded across the thread boundary so the worker handler can bail gracefully. Database/Redis pools cannot be shared across threads, so the worker file boots its own. I/O-bound tasks should stay in-process; this is targeted at sync CPU work like image processing or hashing.
- **Redis fence provider for the task runner** (`svelte-adapter-uws-extensions/redis/fence`). Optional `fence` option on `createTaskRunner` adds a second source of truth for "is this attempt's fence still alive." The Postgres row stays canonical; the provider mirrors the fence value to a Redis key with a short TTL refreshed by heartbeat. On every heartbeat tick the runner consults both sources, and either reporting "lost" aborts the handler. Two atomic Lua scripts back the path: heartbeat is `if get == fence then pexpire end`, release is `if get == fence then del end` - no fence held by another owner can be released or refreshed by accident. Primary value is force-takeover detection (drain an instance, immediately kick its in-flight tasks off without waiting for the Postgres deadline) rather than raw heartbeat throughput.
- **Task runner async path** (`enqueue` + `await` on `createTaskRunner`). `enqueue(name, opts)` inserts a row with `status='pending'` and returns the `taskId` immediately without running the handler. A new dispatch sweep (default every 5s, configurable via `dispatchInterval` / `dispatchBatchSize`) claims pending rows via `FOR UPDATE SKIP LOCKED`, transitions them to `running`, and runs the handler in the background on any live instance with the handler registered. `await(taskId, opts?)` polls the row until terminal and returns the result or throws the stored error; runner-level `awaitPollInterval` / `awaitTimeout` defaults are overridable per call. Use cases: HTTP handlers that respond 202 immediately, cross-instance work distribution where the web tier enqueues and a worker tier processes, fire-and-forget background jobs. Pending rows whose handler is unknown locally are left in `running` for another instance to reclaim via the recovery sweep. Internal cleanup: `runRegisteredTask` was refactored to accept an optional starting fence so dispatch and recovery skip the redundant rearm UPDATE that B2a's recovery path was doing.
- **Replay buffers: `gap(topic, lastSeenSeq)` probe** on both Redis and Postgres backends. Returns `{ truncated, missingFrom }` so a consumer can decide between an incremental `since()` fetch and a full reload without driving a WebSocket replay. The same truncation logic that the existing `replay()` method uses internally is now callable: a non-zero `lastSeenSeq` whose successor is no longer in the buffer (either the buffer was trimmed past it, or the buffer is empty and the seq counter has advanced) returns `truncated: true` with `missingFrom = lastSeenSeq + 1`; otherwise `truncated: false, missingFrom: null`. `lastSeenSeq` of 0 short-circuits to "not truncated" (a fresh client has no history to lose). The Redis path uses `ZRANGEBYSCORE` from `lastSeenSeq + 1` and skips corrupt entries the same way `replay()` does; the Postgres path uses an indexed `SELECT seq ... WHERE seq >= $2 ORDER BY seq ASC LIMIT 1` lookup on `(topic, seq)`. README also gains an "Aggregate vs broadcast topics" subsection: replay buffers should be sized per-aggregate (`auction:a1b2`, `chat:room-7`), not per broadcast channel, so the size budget reflects a real history window per aggregate and gap detection is actionable.
- **Pub/sub bus: first-class degraded / recovered events** (`systemChannel`, `onDegraded`, `onRecovered` options on `createPubSubBus`). When the bus shares a circuit breaker with the rest of the extensions, the bus now subscribes to the breaker and auto-emits `degraded` / `recovered` events on a configurable system topic (default `'__realtime'`) so connected clients can show a stale-data banner without any user-side wiring. The event payload is `{ at: <epoch ms> }`. Auto-emission is local-only - Redis is what's degraded, so the event reaches local clients via the underlying platform without attempting a relay; each instance reports its own breaker state to its own clients. Set `systemChannel: null` or `false` to disable auto-emission; the `onDegraded` / `onRecovered` callbacks remain available for server-side reactions (logging, alerts) regardless. Replaces the manually wired `breaker.onStateChange` -> `distributed.publish('__system', ...)` pattern that the README previously documented; the manual pattern still works for users who want different topics, custom payloads, or cross-instance forwarding.
- **Circuit breaker: `subscribe(handler)`** on `createCircuitBreaker`. Returns an unsubscribe function and supports multiple listeners simultaneously. The constructor's `onStateChange` keeps working (it is added as an initial listener internally) so existing call sites are unchanged. A throwing listener can no longer break the others - listener errors are caught and swallowed inside `transition()`. This lifts the single-callback constraint that previously blocked layered consumers like the pub/sub bus's degraded-event emitter from coexisting with a user-supplied `onStateChange`.
- **Presence: `metrics()` snapshot** on `createPresence`. Synchronous accessor returning `{ totalOnline, heartbeatLatencyMs, staleCleanedTotal }`. `totalOnline` sums unique-users-per-topic across topics this instance is locally tracking. `heartbeatLatencyMs` is the duration of the most recent heartbeat tick, useful as a rough Redis-health indicator. `staleCleanedTotal` is the cumulative count of stale fields removed by the heartbeat-driven cleanup script since startup. The same numbers are exposed as Prometheus gauges (`presence_total_online{topic}`, `presence_heartbeat_latency_ms`) when a metrics registry is attached. The heartbeat tick now always observes the cleanup script result so the internal counter stays accurate even when no Prometheus registry is wired.
- **Presence: `keyspaceNotifications: true` mode** on `createPresence`. Optional opt-in that `psubscribe`-s to `__keyevent@*__:expired` so a sync-only observer of a topic gets an empty `list` event the moment that topic's presence hash expires in Redis - catches the instance-died scenario where the only tracker crashed and the broadcast `leave` event never fired. Requires `CONFIG SET notify-keyspace-events Ex` (or any flagset including `K`/`E` and `x`); `psubscribe` failure is logged once and the rest of the tracker keeps working without the keyspace branch. Scope is hash-key expiry only; per-field expiry of stale entries continues to run via the existing heartbeat cleanup script. Per-field hash TTLs via Redis 7.4 `HEXPIRE` are deliberately out of scope (different storage layout, future item).
- **Replay buffer: `publishIdempotent` (stream backend only)**. Caller-provided idempotency via `{ producerId, requestId }`. Implemented as a Lua-atomic `HGET` (cache lookup) + `INCR` + `XADD` + `HSET` - on a repeat tuple within `idempotencyTtl` (default 48 hours), returns the cached seq, skips the XADD, and skips the local broadcast. The seq counter only advances on fresh writes so duplicate retries do not trigger false-positive truncation. Dedup cache is keyed `{prefix}replay:idmp:{producerId}:{topic}`. Pairs with the durable task runner: a task that publishes to replay can pass its task id as `requestId` so worker-crash retries don't double-publish. Native Redis 8.6+ `XADD IDMP` was evaluated and rejected for our seq-counter architecture (pre-INCR before XADD wastes seqs on duplicates and causes false-positive truncation; the manual Lua approach has no such waste).
- **Replay buffer: `storage: 'stream'` backend** on `createReplay`. Opt-in dispatch to a Redis Streams implementation (`XADD`/`XRANGE` with `<seq>-0` IDs) parallel to the default sorted-set one. Same external contract - same `publish` / `seq` / `gap` / `since` / `replay` / `clear` methods, same `durability: 'replicated'` mode, same metrics. Listpack encoding is more compact than sorted-set encoding, and `XRANGE` against `(seq-0` filters natively by sequence number. Both backends use the same seq counter but different buf-key prefixes (`replay:buf:` vs `replay:streambuf:`) so they can coexist on the same Redis without WRONGTYPE collisions. No built-in migration helper between backends; a single topic should pick one and stay there.
- **Postgres job queue** (`svelte-adapter-uws-extensions/postgres/jobs`). `createJobQueue(pgClient)` is a minimal `SELECT ... FOR UPDATE SKIP LOCKED` queue for vanilla Postgres 9.5+ - no extensions required. `enqueue(queue, payload)` / `claim(queue, { batchSize, visibilityTimeoutMs })` / `complete(idOrIds)` / `fail(idOrIds)` / `extend(idOrIds, ms)` / `pending(queue?)` / `clear(queue?)`. Visibility timeout means a worker that crashes mid-processing has its claim auto-expire so another worker can pick the job up. Max-attempts and dead-letter behavior are deliberately NOT baked in - the `attempts` counter is exposed on every claim, callers track it and decide when to give up. Auto-creates the `svti_jobs` table with two partial indexes (queue+id where unclaimed, claimed_until where claimed). Pairs with `createTaskRunner` as the lighter "ingest event, defer work" producer.
- **Redis Functions wrapper** (`svelte-adapter-uws-extensions/redis/functions`). `createFunctionLibrary(client, code)` parses the library name from the `#!lua name=<libname>` shebang and exposes `load()` (FUNCTION LOAD REPLACE), `call(funcName, { keys, args })` (FCALL), and `delete()` (FUNCTION DELETE). Versioned, hot-reloadable server-side scripts: roll forward without an app deploy. Requires Redis 7+; activate runs `INFO server` and throws on older servers (no EVALSHA fallback - maintaining each function in two forms is more cost than benefit; on Redis 6 use `redis.eval` directly).
- **Sharded pub/sub bus** (`svelte-adapter-uws-extensions/redis/sharded-pubsub`). `createShardedBus` is the SPUBLISH/SSUBSCRIBE variant of `createPubSubBus` for Redis Cluster deployments with many fine-grained topics. Per-topic channels, dynamic subscription via `follow(topic)` / `unfollow(topic)` (or `bus.hooks` against WebSocket subscribe/unsubscribe), refcounted SSUBSCRIBE on first follower per channel and SUNSUBSCRIBE on last out. Optional `shardKey` groups topics into the same channel (e.g. `(topic) => topic.split(':')[0]`). Requires Redis 7+; activate runs `INFO server` and throws on older servers (use `createPubSubBus` for Redis 6 / older Valkey). Most apps still want the simpler `createPubSubBus` - the sharded bus is for cluster + narrow-audience topics where bandwidth saving outweighs the management overhead.
- **Redis replay: `durability: 'replicated'` mode** on `createReplay`. After each write, runs `WAIT minReplicas replicationTimeoutMs`; throws `ReplicationTimeoutError` and skips the local broadcast when fewer replicas ack within the timeout. The data is on the master regardless of outcome (other instances doing `replay()` see it); only the local broadcast is suppressed so live consumers aren't committed to state that could be lost if the master fails before replicas catch up. WAIT command errors bubble up as breaker failures; under-acked timeouts do not trip the breaker - it's a separate signal layer.
- **LISTEN/NOTIFY: leader-elected single-listener mode** (`multiListener: 'advisory'` on `createNotifyBridge`). Each replica polls `pg_try_advisory_lock(lockId)` on a dedicated connection; the winner holds the LISTEN connection and forwards notifications, others stay idle followers. On leader connection loss the session-scoped lock auto-releases and another replica picks up on its next poll. The leader publishes *with* relay so a cross-instance pub/sub bus fans out to non-leader replicas; the existing default (`multiListener: 'all'`) keeps publishing with `relay: false` because every replica has its own LISTEN. Avoids N LISTEN connections in an N-replica deployment.
- **Admission control** (`svelte-adapter-uws-extensions/admission`). Pressure-aware companion to the circuit breaker. `createAdmissionControl({ classes })` accepts a per-class block rule (either a list of pressure reasons or a predicate function) and exposes `shouldAccept(className, platform)`, which reads `platform.pressure` from the adapter and returns `false` when the class should be shed. Where the breaker answers "is the backend up?", admission control answers "are we OK to take more work right now?" - using the worker-local memory / publish-rate / subscriber-ratio signals to gate non-critical work before it reaches a backend. Requires `svelte-adapter-uws >= 0.5.0-next.1` for the `platform.pressure` getter. The reason-precedence math (memory > publish rate > subscribers) lives in the adapter; this controller just maps the resolved reason to a per-class accept/reject decision. Optional Prometheus metrics: `admission_accepted_total{class}` and `admission_rejected_total{class, reason}` track admission decisions for ops dashboards.

### Performance

- **Presence join: stringify userData once and reuse for the local cache.** The validation pass that called `JSON.stringify(data)` and discarded the result now keeps the serialized output for the topic-data cache write. About 25% off the per-call cost of the join hot path on a typical payload.
- **Sorted-set replay: `replay(ws, topic, sinceSeq)` no longer fetches the entire buffer.** Replaced `ZRANGEBYSCORE -inf +inf` plus a double scan with a single pipeline of `ZRANGE 0 9` (oldest-entry probe, 10-entry head so corrupt leaders still surface a valid oldest seq) plus `ZRANGEBYSCORE sinceSeq+1 +inf` (only what the client missed). Same single round-trip. About 99% faster on the common case (client behind by a few of N buffered messages), neutral when the client missed everything. Network bandwidth now scales with miss size rather than buffer size.
- **Cursor and presence snapshot loops: `for (const key of Object.keys(all))`** in place of `Object.entries(all)`. Drops the per-iteration `[k, v]` tuple allocation. 20 to 30 percent faster on hgetall snapshot iteration.
- **Sharded pub/sub `close` hook: `Promise.all` parallel unfollow.** Sequential await per followed topic was a needless serialization on disconnect. Each unfollow's synchronous portion is atomic across the await point so concurrent calls do not race.
- **Presence: reverse `topic|key` index for the leave-time scan.** The two leave paths used to walk every ws on the instance to find a still-alive sibling connection for the same user; they now consult a `Map<topic|key, Set<ws>>` that is maintained on every join and leave. Microbench showed the per-join overhead is about 30 ns regardless of N while the per-leave saving grows with N (1 µs at 100 ws, 144 µs at 10 000 ws), so even a busy instance breaks even after a few dozen joins per leave.
- **Cursor reconnect snapshot fetch: explicit `redis.pipeline()`.** The previous code fired N independent `redis.hgetall(...)` promises and relied on ioredis auto-pipelining; auto-pipelining is off by default. Real-Redis bench against the integration stack measured 11 to 44 percent faster across topic counts 10 to 500 with auto-pipelining off, and roughly neutral with it on.

### Migration

Existing deployments with default table names need to either:

1. **Override the `table` option** on each module to keep the old `ws_*` names: `createReplay(pg, { table: 'ws_replay' })`, etc. Zero-downtime, no SQL needed.
2. **Rename tables in place** before deploying:
   ```sql
   ALTER TABLE ws_replay RENAME TO svti_replay;
   ALTER TABLE ws_replay RENAME COLUMN id TO svti_replay_id;
   ALTER TABLE ws_replay RENAME COLUMN created_date TO created_at;
   ALTER TABLE ws_replay_seq RENAME TO svti_replay_seq;
   ALTER TABLE ws_idempotency RENAME TO svti_idempotency;
   ALTER TABLE svti_idempotency RENAME COLUMN key TO svti_idempotency_key;
   ALTER TABLE ws_jobs RENAME TO svti_jobs;
   ALTER TABLE svti_jobs RENAME COLUMN id TO svti_jobs_id;
   ALTER TABLE ws_tasks RENAME TO svti_tasks;
   ALTER TABLE svti_tasks RENAME COLUMN task_id TO svti_tasks_id;
   ALTER TABLE svti_tasks RENAME COLUMN idempotency_key TO svti_idempotency_key;
   ```

The JS API for `enqueue()` / `claim()` / `await()` results is unchanged (SQL aliases preserve the field names callers read from rows). The only JS-level source change is the `idempotency.acquire(key)` parameter, now `idempotency.acquire(idempotencyKey)` - a function-argument rename, transparent at call sites.

## [0.4.0] - 2025-03-19

### Breaking Changes

#### Replay (Redis and Postgres)

- **`replay()` end event data changed from `null` to `{ reqId }`.** Previously the end marker sent `null` as its data. Now sends `{ reqId: undefined }` (or `{ reqId: 'some-id' }` when a correlation ID is passed). **Action:** update any client-side check from `data === null` to an object check.
- **`replay()` signature accepts optional `reqId` parameter.** `replay(ws, topic, sinceSeq, platform, reqId?)`. Not breaking if you don't pass it, but the wire format above is.
- **New `truncated` event on `__replay:{topic}`.** Sent before replay messages when the buffer has been trimmed past the client's `sinceSeq`. **Action:** handle or ignore this new event in your replay store.

#### Presence

- **Default `select` strips `__`-prefixed and sensitive keys.** Previously the identity function. Now strips keys starting with `__` (e.g. `__subscriptions`, `remoteAddress` injected by core v0.4.0) and keys matching `/token|secret|password|auth|session|cookie|jwt|credential/i`. **Action:** if you relied on these fields in presence data, pass an explicit `select` function.
- **`hooks` now includes `unsubscribe`.** Hooks destructure should become `export const { subscribe, unsubscribe, close } = presence.hooks`. Not strictly breaking (a missing hook is never called), but required for correct single-topic leave on unsubscribe.

#### Cursor

- **Default `select` strips `__`-prefixed and sensitive keys.** Same behavior change as presence above.

#### Groups

- **JOIN_SCRIPT returns array `[status, ...liveMembers]` instead of single integer.** Internal change but affects anyone calling the Lua script directly.

#### Package

- **Peer dependency changed from `svelte-adapter-uws >=0.2.0` to `>=0.4.0`.** Core adapter must be upgraded first.

### Added

#### New Modules

- **Prometheus metrics** (`svelte-adapter-uws-extensions/prometheus`). Zero-dependency metrics registry in Prometheus text exposition format. Zero overhead when not enabled - every extension uses optional chaining on a nullish reference. Includes `createMetrics()`, counter/gauge/histogram primitives, `mapTopic` for cardinality control, and a built-in uWS HTTP handler.
- **Circuit breaker** (`svelte-adapter-uws-extensions/breaker`). Three-state breaker (healthy/broken/probing) that prevents thundering herd when a backend goes down. All extensions accept an optional `breaker` option. Awaited operations fail fast; fire-and-forget operations (heartbeat, relay, cursor broadcast) are skipped entirely.
- **Shared utilities** - `shared/time.js` (cached `Date.now()`), `shared/scripts.js` (deduplicated Lua scripts), `shared/errors.js` (`ConnectionError`, `TimeoutError`).

#### All Extensions

- `metrics` option on every extension for Prometheus instrumentation.
- `breaker` option on every extension for circuit breaker integration.

#### Cursor

- `snapshot(ws, topic, platform)` - send all current cursor positions to a single connection.
- `hooks` helper - ready-made `subscribe`, `message`, `close` hooks for zero-config cursor tracking.

#### Groups

- `hooks` helper - ready-made `subscribe`, `unsubscribe`, `close` hooks for zero-config group membership.

#### Presence

- `unsubscribe` hook for single-topic leave when the client unsubscribes (requires core v0.4.0+).
- `updated` event - broadcast when a user's data changes (deep equality check), instead of silently overwriting.

#### Pub/Sub

- Wrapped platform now relays `batch()` calls to Redis, not just `publish()`.

#### Rate Limiting

- Atomic `ban()` via Lua script - single round trip instead of multi-step `hget` + `hmset`.

### Changed (Under the Hood)

#### Performance

- Presence heartbeat, cursor broadcast, and groups heartbeat use Redis pipelines instead of individual commands.
- PubSub relay batching via microtask - multiple publishes within one tick coalesced into a single pipelined round trip.
- Presence `list()` and `count()` use server-side Lua scripts (`LIST_SCRIPT`, `COUNT_DEDUP_SCRIPT`) - avoids transferring the full hash to Node.
- Groups `count()` uses server-side Lua (`COUNT_SCRIPT`).
- Cached `Date.now()` via `shared/time.js` for hot paths (heartbeats, throttle checks, staleness).
- Replay Lua avoids double `cjson.encode` - builds JSON payload via string concat.
- Rate limit Lua uses `hmget` (one round trip) instead of 3x `hget`; `hset` instead of deprecated `hmset`.
- Postgres replay publish uses a CTE to combine sequence increment + insert into one query.
- Postgres replay trim is sequence-based (O(1) cutoff calculation) instead of `COUNT(*)` + subquery.
- Postgres replay uses prepared (named) statements.
- Postgres periodic cleanup query rewritten to use `OFFSET`-based cutoff instead of `ROW_NUMBER()` window function.
- `redis.del()` replaced with `redis.unlink()` across all `clear()` methods (non-blocking delete).
- Lua scripts deduplicated into `shared/scripts.js`.

#### Reliability

- Presence join has full rollback (`undoJoin()`) that reverts local state, Redis writes, and compensating events on any failure during the async flow.
- Groups join rolls back the Redis member entry if `ws.subscribe()` fails after the Lua insert.
- `ws.unsubscribe()` wrapped in try/catch throughout groups, presence, and cursor for already-closed connections.
- Presence subscriber auto-closes after 30s of no active topics (idle timeout).
- Cursor cleanup timer is lazy - starts on first topic, stops when last topic is removed.
- Presence and cursor filter out timestamp-less entries as stale (previously only filtered if timestamp was present and expired).
- Redis client `duplicates` tracking uses `Set` with auto-cleanup on `close`/`end` events.
- Postgres replay cleanup guarded against concurrent runs.

#### Validation and Safety

- Input validation on all construction options (`reconnectInterval`, `throttle`, `ttl`, `memberTtl`, `heartbeat`, `select`, `parse`, etc.).
- `select()` results validated as JSON-serializable at join/update time.
- Sensitive data warnings - presence and cursor warn once if userData contains keys matching `token`, `secret`, `password`, etc.
- Notify bridge warns when payload approaches Postgres ~8000 byte NOTIFY limit.
